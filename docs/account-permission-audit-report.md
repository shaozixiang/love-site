## 一、这次主要解决的问题

你反馈的问题是：

1. 管理员注册的新账号，在别的手机/电脑上登录不了。
2. 后台里面设置的权限有时候不生效。
3. 注册、改权限、重置密码这些操作，看起来成功了，但实际可能没有同步到云端。
4. Cloudflare 部署时可能还在用旧的 `index.html`，而不是最新的 `index.txt`。

我检查后发现，核心原因不是 GitHub 图片上传，而是账号系统以前太依赖本地浏览器缓存。

简单说：以前网站有时候会先把账号保存到当前浏览器的 localStorage，然后再慢慢同步到 Supabase。如果 Supabase 表缺字段、没有唯一约束、权限规则不允许写入，当前设备看起来像“注册成功”，但别的设备根本读不到这个账号。

## 二、我改了哪些代码

### 1. 强制登录从 Supabase 云端读取账号

修改位置：`index.txt` 和同步后的 `index.html`

现在登录时不会再优先相信本地缓存。

也就是说：

- 如果账号真的在 Supabase 的 `users` 表里，其他设备就能登录。
- 如果 Supabase 读不到账号，页面会明确提示“无法从 Supabase 读取账号”。
- 这样不会再出现“这台电脑能用，换个手机就没有这个账号”的假成功情况。

### 2. 管理员注册账号必须同步到云端才算成功

以前注册新账号可能本地保存了，但云端失败也不明显。

现在注册账号时：

- 会写入完整字段：密码、头像、权限、角色。
- 默认权限是 `full`，默认角色是普通用户 `user`。
- 如果 Supabase 保存失败，会弹窗告诉你失败原因。
- 不会让你误以为账号已经注册成功。

### 3. 权限检查改成实时读取云端

以前权限判断可能读到旧缓存，所以后台改了权限，当前设备不一定马上生效。

现在每次做关键操作前都会重新检查云端权限：

- `view`：只能看。
- `comment`：可以评论、点赞、提交道歉。
- `full`：可以发布内容、上传图片、添加行程/地图/待办、删除自己的内容。
- `admin`：管理员，不受普通权限限制。

### 4. 管理后台操作重新校验管理员身份

以前只要页面里的 `isAdmin` 状态没刷新，可能会出现按钮显示不准、权限判断不准。

现在这些操作都会先去 Supabase 重新确认是不是管理员：

- 打开后台。
- 注册新账号。
- 删除账号。
- 切换管理员/普通用户。
- 设置权限。
- 重置密码。

如果云端确认不了管理员身份，会直接提示错误，不会继续操作。

### 5. 管理后台按钮显示修复

以前如果同一个浏览器先登录管理员，再切换普通账号，管理员按钮可能显示状态不干净。

现在进入系统和权限刷新时都会执行：

- 管理员：显示“管理后台”。
- 普通用户：隐藏“管理后台”。

### 6. 重置密码失败会明确提示

以前 `resetUserPwd(username)` 没有完整的失败处理。

现在如果重置密码没有成功写入 Supabase，会提示：

> 重置密码失败：账号没有同步到云端。

这样不会出现当前页面显示成功、其他设备还是旧密码的情况。

### 7. 评论删除权限补上异步检查

我修了两个评论删除点：

- 相册动态评论删除。
- 留言板评论删除。

现在删除评论前会真正等待权限检查完成，不会跳过权限判断。

### 8. 气泡照片设置入口加了权限检查

以前点击右上角气泡设置入口时没有在打开前强制检查完整权限。

现在打开共享气泡照片池也需要 `full` 权限。

### 9. `index.html` 已同步为最新源码

Cloudflare Pages 部署静态站通常读取 `index.html`。

我已经把：

- `index.txt`
- `index.html`

同步成完全一样的内容。

也就是说，你之后部署 Cloudflare 时用 `index.html` 就是最新版本。

## 三、你还需要做的一步：运行 Supabase SQL

这一步很重要。如果不做，权限功能和跨设备账号同步仍然可能失败。

请打开：

`C:\Users\邵子祥\Desktop\love\docs\supabase-permissions.sql`

然后：

1. 打开 Supabase 后台。
2. 进入左侧 `SQL Editor`。
3. 点 `New query`。
4. 把 `supabase-permissions.sql` 里面的内容全部复制进去。
5. 点 `Run`。

这个 SQL 会做几件事：

- 给 `users` 表补 `permission` 字段。
- 给 `users` 表补 `role` 字段。
- 给 `users` 表补在线状态字段。
- 给 `username` 加唯一限制，保证同一个账号不会重复出现。
- 修正旧数据里面空的权限和角色。

如果 SQL 报错说有重复 username，意思是 `users` 表里有同名账号。你需要先手动删掉重复的，只保留一条，再重新运行 SQL。

## 六、你部署后怎么测试

建议你按这个顺序测试：

### 1. 管理员登录

用管理员账号登录网站。

如果登录失败，先检查 Supabase 的 `users` 表里有没有 `admin` 这一行。

### 2. 注册一个测试账号

后台注册一个新账号，比如：

- 账号：`test01`
- 密码：`123456`

注册成功后，打开 Supabase 的 `users` 表，确认能看到 `test01`。

如果 Supabase 表里看不到，说明不是前端显示问题，而是数据库写入失败。

### 3. 用另一台设备登录

拿手机或另一个浏览器登录：

- 账号：`test01`
- 密码：`123456`

如果能登录，说明跨设备账号同步已经正常。

### 4. 测试权限

回到管理员后台，把 `test01` 分别设置成：

- `view`：只能浏览，不能点赞评论，不能发布。
- `comment`：可以评论点赞，但不能发动态和上传图片。
- `full`：可以发布内容和上传图片。

每改一次，去另一台设备刷新或重新操作一下，看限制是否生效。

### 5. 测试图片上传

用 `full` 权限账号上传一张新图。

正常结果：

- GitHub 仓库 `couple-images` 里能看到图片。
- 网站里图片能显示。
- Supabase 数据库里只保存图片链接，不存图片文件本身。

## 七、关于以前数据库里的旧照片迁移

不麻烦，但建议单独做，不要和这次权限修复混在一起。

迁移旧照片大概流程是：

1. 从 Supabase 表里读取旧图片 URL。
2. 一个一个下载旧图片。
3. 上传到 GitHub 仓库。
4. 得到新的 jsDelivr CDN 链接。
5. 先生成一张“旧链接 -> 新链接”的对照表。
6. 确认新链接都能打开后，再批量更新 Supabase 里的 URL。

这件事可以做成一个单独脚本，风险不大，但要小心：不要一上来直接覆盖旧链接，最好先备份。

## 八、这次修改后的结论

这次已经把账号和权限系统从“本地缓存优先”改成了“云端 Supabase 为准”。

以后判断一个账号有没有注册成功，标准只有一个：

> Supabase 的 `users` 表里能看到这个账号。

> Supabase 的 `users.permission` 和 `users.role` 字段是什么，网站就按什么执行。
